chapitre 1

Premiers pas avec Git

Git : le guide est le dernier volet d'une trilogie de tutoriels sur les fondamentaux du développement, conçus pour enseigner trois compétences essentielles aux développeurs de logiciels et à ceux qui travaillent avec eux. Les seules conditions préalables sont les deux premiers tutoriels de la trilogie,
Ligne de commande: l'éssentiel (qui couvre la ligne de commande Unix) et Visual Studio Code: l'éssentiel (qui couvre l'éditeur de texte le plus utilisé au monde vs code).

Ce tutoriel couvre une troisième compétence essentielle : le contrôle de version. Comme ses deux prédécesseurs, Git : le guide ne suppose pas que vous soyez familier avec cette catégorie d'applications. Si vous ne savez pas exactement ce qu'est le « contrôle de version », vous êtes au bon endroit. Même si vous connaissez déjà le sujet, vous apprendrez probablement beaucoup de choses dans ce tutoriel. Quoi qu'il en soit, l'apprentissage du contenu de «Git : le guide » vous prépare aux autres tutoriels « Sur les # langages de programmation ».

Encadré 1.1. Les vrais artistes livrent
Comme l'a dit un jour le légendaire cofondateur d'Apple, Steve Jobs : « Les vrais artistes livrent ». Ce qu'il voulait dire, c'est que même s'il est tentant de peaufiner son travail en privé à l'infini, les créateurs doivent livrer leur travail, c'est-à-dire le terminer et le diffuser dans le monde. Cela peut être effrayant, car livrer signifie exposer son travail non seulement à ses fans, mais aussi à ses détracteurs. « Et si les gens n'aiment pas ce que j'ai fait ? » Les vrais artistes livrent. Il est important de comprendre que la livraison est une compétence distincte de la création. De nombreux créateurs deviennent doués pour créer des choses, mais n'apprennent jamais à les livrer. Pour éviter que cela ne nous arrive, dans ce tutoriel, nous allons livrer deux choses : un dépôt Git public et un bonus surprise qui vous donnera le droit de vous vanter auprès de tous vos amis.

Le contrôle de version résout un problème qui vous est peut-être familier si vous avez déjà vu des documents Word ou des feuilles de calcul Excel portant des noms tels que Report_2014_1.doc, Report_2014_2.doc, Report_2014_3.doc ou budget-v7.xls. Ces noms encombrants montrent à quel point il peut être fastidieux de suivre les différentes versions d'un document. De nos jours, des applications telles que Word proposent parfois un suivi des versions intégré, mais ces fonctionnalités sont étroitement liées à l'application sous-jacente et ne sont pas utiles pour d'autres types de documents. De nombreuses applications techniques (y compris la plupart des sites web et des projets de programmation) nécessitent une solution générale au problème des versions.

Un système de contrôle de version, ou VCS, permet de suivre automatiquement les modifications apportées aux projets logiciels, donnant aux créateurs la possibilité de consulter les versions précédentes des fichiers et des répertoires, de développer des fonctionnalités spéculatives sans perturber le développement principal, de sauvegarder en toute sécurité le projet et son historique, et de collaborer facilement et commodément avec d'autres personnes. De plus, l'utilisation du contrôle de version facilite considérablement le déploiement de sites web et d'applications web de production. Par conséquent, la maîtrise d'au moins un système de contrôle de version est un élément essentiel de la sophistication technique (encadré 1.2), une compétence utile pour les développeurs, les concepteurs et les gestionnaires. Cela s'applique en particulier au système de contrôle de version présenté dans ce tutoriel, appelé Git.

Encadré 1.2. Sophistication technique
L'un des thèmes principaux des tutoriels développement de la sophistication technique, c'est-à-dire la combinaison de compétences techniques et relationnelles qui donnent l'impression que vous pouvez résoudre comme par magie n'importe quel problème technique. Elle est trés important pour développer ces compétences, car la maîtrise d'au moins un système de contrôle de version moderne est un élément essentiel de la sophistication technique. Dans le contexte de Git, la sophistication technique comprend plusieurs éléments. De nombreuses commandes Git affichent divers détails sur l'écran du terminal ; la sophistication technique vous permet de déterminer ceux auxquels vous devez prêter attention et ceux que vous pouvez ignorer. Il existe également de nombreuses ressources liées à Git sur le Web, ce qui signifie entre autres que les recherches Google sont souvent utiles pour trouver la commande exacte dont vous avez besoin à un moment donné. La sophistication technique vous permet de déterminer les meilleurs termes de recherche pour trouver la réponse que vous cherchez ; par exemple, si vous devez supprimer une branche distante (section 4.3.1), rechercher « git delete remote branch » sur Google est un bon moyen de trouver quelque chose d'utile. Enfin, les sites d'hébergement de référentiels tels que GitHub, GitLab et Bitbucket comprennent généralement des commandes qui vous guident dans diverses tâches de configuration, et la maîtrise technique vous donne la confiance nécessaire pour suivre les étapes même si vous ne comprenez pas tous les détails. Une commande utile pour apprendre Git est git help, qui fournit à elle seule des directives générales sur l'utilisation de Git et, lorsqu'elle est appliquée à une commande spécifique, donne des informations supplémentaires sur cette commande. Par exemple, git help add affiche des détails sur la commande git add. La sortie de git help est similaire aux pages man couvertes dans pleine d'informations utiles mais souvent obscures. Comme toujours, utilisez votre sophistication technique pour vous aider à les comprendre.

Le contrôle de version a considérablement évolué au fil des ans. La lignée qui a conduit à Git comprend des programmes appelés RCS(Revision Control System 1982 un des premiers ensemble de code unix permettant à plusieurs utilisateurs de developper et gerer du code), CVS(Concurrent Versioning System 1986 Dick Grune interface pour rcs, il s'appuie dessus auquel il ajoute le suivi des modifications) et Subversion(en 2000 est un système de contrôle de version distribué en open source sous la licence Apache . [ 1 ] Les développeurs de logiciels utilisent Subversion pour gérer les versions actuelles et historiques de fichiers tels que le code source , les pages web et la documentation. Son objectif est de succéder, dans la mesure du possible, au système de gestion de versions concurrentes), et il existe également de nombreuses alternatives actuelles, notamment Perforce, Bazaar et Mercurial. Je mentionne ces exemples non pas parce que vous devez les connaître, mais uniquement pour montrer la diversité déconcertante qui existe. Pire encore, lorsque vous choisissez un système de contrôle de version, vous vous engagez vraiment à l'utiliser1, et il est souvent difficile de passer d'un système à l'autre. Heureusement, ces dernières années, un vainqueur incontesté s'est imposé dans la guerre des VCS open source : Git. Cette victoire est la principale raison pour laquelle ce tutoriel s'intitule «Git: le guide » plutôt que «contrôle de version ». Néanmoins, bon nombre des idées présentées ici sont assez générales, et si par hasard vous devez utiliser un autre VCS, ce tutoriel vous fournira tout de même une introduction utile au sujet.

Développé à l'origine par le créateur de Linux, Linus Torvalds, pour héberger le noyau Linux, Git est un programme en ligne de commande conçu dans la tradition Unix (c'est pourquoi une bonne connaissance de la ligne de commande Unix est une condition préalable importante). Git combine puissance, rapidité et adoption par la communauté, ce qui lui laisse peu de rivaux, mais il peut être difficile à apprendre, et les autres tutoriels Git ont tendance à présenter beaucoup de théorie complexe, qui peut être intéressante à apprendre mais qui, dans la pratique, n'est vraiment comprise que par une poignée d'utilisateurs Git (comme l'illustre « Git » via la bande dessinée en ligne xkcd). La bonne nouvelle, c'est que l'ensemble des commandes Git nécessaires pour être productif est relativement restreint ; vous trouverez quelques références à des ressources plus avancées et plus théoriques dans la section 4.7, mais dans ce tutoriel, nous nous concentrons sur les commandes essentielles .

Encadré 1.3. Passer macOS à Bash
Si vous utilisez macOS, vous devez à ce stade vous assurer que vous utilisez le bon programme shell pour ce tutoriel. Le shell par défaut de macOS Catalina est Z shell (Zsh), mais pour obtenir des résultats cohérents avec ce tutoriel, vous devez passer au shell appelé Bash. La première étape consiste à déterminer quel shell votre système utilise, ce que vous pouvez faire à l'aide de la commande echo :

$ echo $SHELL
/bin/bash

Cela affiche la variable d'environnement $SHELL. Si vous voyez le résultat ci-dessus, indiquant que vous utilisez déjà Bash, vous avez terminé et pouvez passer à la suite du tutoriel. (Dans de rares cas, $SHELL peut différer du shell actuel, mais la procédure ci-dessous permettra tout de même de passer correctement d'un shell à un autre.) Pour plus d'informations, notamment sur la manière de passer au shell Z et de l'utiliser avec ce tutoriel, consultez l'article du blog les tutoriels ». L'autre résultat possible de la commande echo est le suivant :

$ echo $SHELL
/bin/zsh

Si tel est le résultat obtenu, vous devez utiliser la commande chsh (« changer de shell ») comme suit :

$ chsh -s /bin/bash

À ce stade, vous serez très certainement invité à saisir le mot de passe de votre système, ce que vous devez faire. Quittez ensuite complètement votre programme shell à l'aide de la commande Command-Q et relancez-le. Vous pouvez vérifier que la modification a bien été effectuée à l'aide de la commande echo :

$ echo $SHELL
/bin/bash

À ce stade, vous verrez probablement apparaître l'alerte suivante, que vous devez ignorer :

Le shell interactif par défaut est désormais zsh. Pour mettre à jour votre compte afin d'utiliser zsh, veuillez exécuter `chsh -s /bin/zsh`. Pour plus de détails, veuillez consulter https://support.apple.com/kb/HT208050. [~]$

Notez que la procédure ci-dessus est entièrement réversible, vous n'avez donc pas à craindre d'endommager votre système. Pour plus d'informations, consultez « Utilisation de Z Shell sur Mac avec les tutoriels ».

1.1 Installation et configuration

La manière la plus courante d'utiliser Git est via un programme en ligne de commande appelé git, qui nous permet de transformer un répertoire Unix ordinaire en un dépôt (ou repo en abrégé) qui nous permet de suivre les modifications apportées à notre projet.3 Dans cette section, nous commencerons par installer Git (si nécessaire) et effectuerons quelques configurations ponctuelles.

Avant toute chose, nous devons d'abord vérifier si Git est installé sur le système actuel. Pour rappel, nous travaillons dans la tradition Unix, il est donc fortement recommandé d'utiliser macOS ou Linux. Les utilisateurs de Microsoft Windows sont encouragés à configurer un environnement de développement compatible avec Linux en suivant les étapes Windows décrites dansle tutoriell'environnement de développement ou à utiliser un IDE cloud basé sur Linux (également décrit dansle tutoriell'environnement de développement ).

La façon la plus simple de vérifier la présence de Git est de lancer une fenêtre de terminal et d'utiliser which4 dans la ligne de commande pour voir si l'exécutable git est déjà présent :

$ which git
/usr/local/bin/git

Si le résultat est vide ou si le message indique que la commande est introuvable, cela signifie que vous devez installer Git manuellement. Pour ce faire, suivez les instructions de la section « Getting Started – Installing Git » (Pour commencer – Installation de Git) de la documentation officielle de Git. (Cela vous donnera probablement l'occasion d'appliquer certaines techniques sophistiquées (encadré 1.2).)

L'étape suivante consiste à vous assurer que vous disposez d'une version suffisamment récente de Git, ce que vous pouvez vérifier comme suit :

$ git --version
git version 2.31.1 #doit être au moins 2.28.0

si la version n'est pas assez récente, vous devez soit installer Git en suivant les instructions de la section « Pour commencer – Installation de Git », soit suivre l'une des suggestions ci-dessous :

*Cloud IDE : si vous utilisez le Cloud IDE recommandé dans «le tutoriell'environnement de développement », exécutez la commande du listing 1.1.

*macOS : si vous utilisez macOS, exécutez la commande du listing 1.2. (Si Homebrew n'est pas installé, suivez d'abord les instructions pour l'installer.)

Liste 1.1 : Mise à niveau de Git sur l'IDE cloud.
$ source <(curl -sL https://cdn.learnenough.com/upgrade_git)

Liste 1.2 : Mise à niveau de Git sur macOS.
$ brew upgrade git

Après avoir installé Git, mais avant de démarrer un projet, nous devons effectuer quelques étapes de configuration uniques, comme indiqué dans le listing 1.3. Il s'agit de configurations globales, ce qui signifie que vous ne devez les effectuer qu'une seule fois par ordinateur. (Ne vous souciez pas de la signification ou de la structure de ces commandes à ce stade.)

Liste 1.3 : Paramètres de configuration globaux uniques.
$ git config --global user.name « Votre nom »
$ git config --global user.email your.email@example.com
$ git config --global init.defaultBranch main

Les deux premiers paramètres de configuration permettent à Git d'identifier les modifications apportées à vos projets par leur nom et leur adresse e-mail, ce qui est particulièrement utile lorsque vous collaborez avec d'autres personnes (chapitre 4). Notez que le nom et l'adresse e-mail que vous utilisez dans le listing 1.3 seront visibles dans tous les projets que vous rendez publics. Ne divulguez donc aucune information que vous préférez garder privée.

La troisième ligne du listing 1.3 définit le nom de la branche Git par défaut sur main, qui est la valeur par défaut actuellement recommandée. Sachez que le nom de branche par défaut pendant les 15 premières années d'existence de Git était master. Vous rencontrerez donc inévitablement de nombreux dépôts Git qui utilisent master au lieu de main.5 (Être capable de gérer ce type de situation est un signe de sophistication technique (encadré 1.2).)

En plus de la configuration du listing 1.3, Git : le guide inclut une configuration avancée facultative (section 4.6) que je vous recommande d'effectuer à un moment donné. Si vous connaissez déjà Git ou si vous êtes un utilisateur expérimenté de la ligne de commande Unix, je vous recommande de suivre les étapes de la section 4.6 dès maintenant, mais sinon, je vous recommande de reporter la configuration avancée à plus tard.

1.2 Initialisation du dépôt

Il est maintenant temps de commencer à créer un projet et de le placer sous contrôle de version avec Git. Pour comprendre le fonctionnement de Git et ses avantages, il est utile d'avoir une application concrète à l'esprit. Nous allons donc suivre les modifications apportées à un projet simple composé d'un petit site web de deux pages, une page d'accueil et une page « À propos ». Nous allons commencer par créer un répertoire avec le nom générique « website » dans un répertoire de dépôts appelé « repos » :

[~]$ mkdir -p repos/website

Nous avons ici utilisé la commande « créer un répertoire » mkdir décrite dansle tutoriella ligne de commande , avec l'option -p, qui permet à mkdir de créer des répertoires intermédiaires si nécessaire (dans ce cas, repos). Notez également que j'ai inclus le répertoire actuel dans l'invite (dans ce cas, [~]) comme prévu par la configuration du listing 4.15. Après avoir créé le répertoire, nous pouvons y accéder comme suit :

[~]$ cd repos/website/
[website]$

(Rappelez-vous que vous pouvez utiliser la complétion par tabulation lorsque vous changez de répertoire. Dans la réalité, je taperais probablement quelque chose comme cd re⇥w⇥.)

Même si le répertoire du site web est vide, nous pouvons déjà le convertir en référentiel, que vous pouvez considérer comme une sorte de répertoire Unix amélioré avec la possibilité supplémentaire de suivre les modifications apportées à chaque fichier et sous-répertoire. Pour créer un nouveau référentiel avec Git, il faut utiliser la commande init (abréviation de « initialize »), qui crée un répertoire caché spécial appelé .git dans lequel Git stocke les informations dont il a besoin pour suivre les modifications de notre projet. (C'est la présence d'un répertoire .git correctement configuré qui distingue un référentiel Git d'un répertoire ordinaire.)

Toutes les commandes Git se composent du programme en ligne de commande git suivi du nom de la commande. La commande complète pour initialiser un dépôt est donc git init, comme indiqué dans le listing 1.4.

Listing 1.4 : Initialisation d'un dépôt Git.
[website]$ git init
Initialisation du dépôt Git vide dans /Users/mhartl/repos/website/.git/
[website (main)]$

L'invite affichée dans le listing 1.4 reflète à la fois la personnalisation de Bash décrite dans « Text Editor pour se perfectionner » et la configuration avancée de la section 4.6.2, votre invite peut donc être différente.7 En particulier, l'invite dans la ligne surlignée du listing 1.4 affiche le nom de la branche Git actuelle, appelée main. Ne vous inquiétez pas pour l'instant de ce que cela signifie ; nous aborderons les branches à partir de la section 3.3.

1.3 Our first commit

Git ne nous permet pas de terminer l'initialisation du référentiel tant qu'il est vide, nous devons donc apporter une modification au répertoire actuel. Nous effectuerons une modification plus importante dans un instant, mais pour l'instant, nous suivrons une convention courante et utiliserons simplement touch pour créer un fichier vide (comme mentionné dans Command Line pour se perfectionner). Dans ce cas, nous créons un site web simple, et la convention quasi universelle consiste à appeler la page principale index.html :

[site web (principal)]$ touch index.html

Après avoir créé ce premier fichier, nous pouvons utiliser la commande git status pour voir le résultat :

site web (principal)]$ git status
Sur la branche principale

Aucun commit pour l'instant

Fichiers non suivis :
(utilisez « git add ... » pour inclure dans ce qui sera commité)
index.html
rien à ajouter au commit mais des fichiers non suivis sont présents (utilisez « git add » pour les suivre)

Nous voyons ici que le fichier index.html est « non suivi », ce qui signifie que Git ne le connaît pas encore. Nous pouvons l'ajouter à l'aide de la commande git add :>

[site web (principal)]$ git add -A

Ici, l'option -A indique à Git d'ajouter tous les fichiers non suivis, même s'il n'y en a qu'un seul dans ce cas. D'après mon expérience, dans 99 % des cas, lorsque vous ajoutez des fichiers, vous souhaitez tous les ajouter. C'est donc une bonne habitude à prendre, et l'apprentissage de l'ajout de fichiers individuels est laissé à titre d'exercice (section 1.3.1). (À propos, la commande git add ., où le point fait référence au répertoire actuel, est également courante.)8

Nous pouvons voir le résultat de git add -A en exécutant à nouveau git status :

[site web (principal)]$ git status
Sur la branche principale
Aucun commit pour l'instant
Modifications à valider :
(utilisez « git rm --cached ... » pour désactiver)
nouveau fichier : index.html

Comme le suggère le mot « désactiver », le statut du fichier est passé de « non suivi » à « activé », ce qui signifie que le fichier est prêt à être ajouté au référentiel. Non suivi/non préparé et préparé sont deux des quatre états couramment utilisés par Git, comme le montre la figure 1.1. (Techniquement, non suivi et non préparé sont des états différents, mais la distinction est rarement importante car git add suit et prépare les fichiers en même temps.)

Figure 1.1 : Séquence principale des états Git pour la modification d'un fichier.

Comme le montre la figure 1.1, après avoir placé les modifications dans la zone de staging, nous pouvons les intégrer au dépôt local en les validant à l'aide de git commit. (Nous aborderons la dernière étape de la figure 1.1, git push, dans la section 2.3.) La plupart des utilisations de git commit utilisent l'option de ligne de commande -m pour inclure un message indiquant l'objectif du commit (encadré 1.4). Dans ce cas, l'objectif est d'initialiser le nouveau dépôt, ce que nous pouvons indiquer comme suit :

[site web (principal)]$ git commit -m « Initialiser le dépôt »
[main (root-commit) 44c52d4] Initialiser le dépôt
1 fichier modifié, 0 insertion(+), 0 suppression(-)
créer le mode 100644 index.html

(J'ai affiché ici mon résultat pour être complet, mais vos détails varieront.)

Encadré 1.4. Valider dans Git De par sa conception, Git exige que chaque validation inclue un message décrivant l'objectif de la validation. En général, ce message prend la forme d'une seule ligne, généralement limitée à environ 72 caractères, avec un message plus long facultatif si vous le souhaitez (section 4.2.3). Bien que les conventions relatives aux messages de validation varient (comme le montre avec humour la bande dessinée xkcd « Git Commit »), le style adopté dans ce tutoriel consiste à rédiger les messages de validation au présent en utilisant le mode impératif, comme dans « Initialiser le dépôt » plutôt que « Initialise le dépôt » ou « Dépôt initialisé ». La raison de cette convention est que Git modélise les commits comme une série de transformations de texte, et dans ce contexte, il est logique de décrire ce que fait chaque commit plutôt que ce qu'il a fait. De plus, cette utilisation est conforme à la convention suivie par les messages de commit générés par les commandes Git elles-mêmes (par exemple, « merge » plutôt que « merges » ou « merged »). Pour plus d'informations, consultez l'article GitHub « Shiny new commit styles ».

[site web (principal)]$ git log

3.2 Ignorer les fichiers

Un problème fréquent lors de la gestion des référentiels Git est de tomber sur des fichiers que vous ne souhaitez pas valider.Il s'agit notamment de fichiers contenant des informations d'identification secrètes, de fichiers de configuration qui ne sont pas partagés entre les ordinateurs, de fichiers temporaires, de fichiers journaux, etc.

Par exemple, sur macOS, un effet secondaire courant de l'utilisation du Finder pour ouvrir des répertoires est la création d'un fichier caché appelé .DS_Store. 3Cet effet secondaire est si courant que les versions plus récentes de Git l'ignorent .DS_Storeautomatiquement, mais nous pouvons simuler un tel effet secondaire en utilisant touchpour créer un exemple .unwanted_DS_Storede fichier comme suit :

[site Web (principal)]$ touch .unwanted_DS_Store

[site web (principal)]$ git status

aucun ajout à la validation, mais des fichiers non suivis sont présents (utilisez « git add » pour les suivre).

C'est ennuyeux, car nous n'avons pas besoin de suivre ce fichier, et en effet, lorsque nous collaborons avec d'autres utilisateurs, cela pourrait facilement provoquer des conflits ( Section 4.2 ) par la suite.

3.3 Branching and merging

L’une des fonctionnalités les plus puissantes de Git est sa capacité à créer des branches , qui sont en fait des copies entièrement autonomes de la source du projet, ainsi que la possibilité de fusionner une branche dans une autre, incorporant ainsi les modifications dans la branche d’origine.L'avantage d'une branche est que vous pouvez apporter vos modifications au projet de manière isolée de la copie principale du code, puis fusionner vos modifications uniquement lorsqu'elles sont terminées.Ceci est particulièrement utile lors de la collaboration avec d'autres utilisateurs ( Chapitre 4 ) ; avoir une branche séparée vous permet d'effectuer des modifications indépendamment des autres développeurs, réduisant ainsi le risque de conflits accidentels.

La manière la plus courante de combiner des branches est git merge, mais il existe une deuxième méthode appelée git rebaseque vous rencontrerez probablement à un moment donné.Mon conseil pour l'instant est : ignorezgit rebase .Les différences entre la fusion et le rebasage sont subtiles et les conventions d'utilisation rebasediffèrent.
